# Apple セキュリティリリース キャッチアップスキル

Apple セキュリティリリース（https://support.apple.com/ja-jp/100100）の新着セキュリティアップデートを取得して、要約付きのMarkdownダイジェストとして保存する。1回の実行につき1つのダイジェストファイルを作成する。

## 実行環境

このスキルはローカルClaude Code（CLI）とデスクトップ版Claudeアプリのコードモードの両方で動作する。利用可能なツールセットが環境ごとに異なるため、Web取得は「優先するツール → フォールバック」を順に試すこと。

- **Web取得**: ブラウザツール（Chrome / Brave 等のMCPサーバ）が利用可能ならそれを最優先。利用不可な場合は `WebFetch` を使う
- **ファイル保存**: 本リポジトリでは `content/catchup/` ディレクトリ配下に保存する

## 最重要: 取得方法の注意

**このソースには公式RSSフィードが存在しない。** 情報源は以下の一覧ページ:

```
https://support.apple.com/ja-jp/100100
```

このページには各セキュリティリリースの **名称（製品・バージョン）・対象（Available for）・公開日（リリース日）・詳細ページへのリンク** がテーブル形式で含まれている。

- **`support.apple.com` は環境のネットワークポリシー次第でアクセスがブロックされる**ことがある。アクセスできない場合は、**このソースをスキップし「support.apple.com にアクセスできなかったため Apple セキュリティリリースはスキップ」と報告して終了する**（エラーで全体を止めない。推測で内容を作らない）
- 各リリースの詳細ページ（例: `https://support.apple.com/ja-jp/124xxx`）はリンクに記載のURLをそのまま使う。URLは推測しない
- URLは `ja-jp` の日本語版を使う

## 絶対に守るべきルール

このスキルで最も重要なのは**正確性**。以下は必ず守ること:

1. **取得したページの内容だけを使う。** 自分の記憶や推測でリリース・要約・日付・CVEを補完しない
2. **URLはページに記載のものを使う。** URLを推測しない
3. **新着リリースを漏れなく拾う。** 既存ファイルと重複しないリリースはすべて対象にする
4. **新着がなければファイルを作らない。** すべて既存の場合は「新しいセキュリティリリースはありませんでした」と報告して終了する
5. **ページにアクセスできなければスキップする。** 取得失敗時に記憶で内容を生成しない
6. **ブラウザツールで開いたタブは必ず閉じる**（ブラウザツールを利用した場合のみ）

## 実行手順

### 1. 一覧ページの取得

`https://support.apple.com/ja-jp/100100` を取得し、掲載されているセキュリティリリースの **名称（製品・バージョン）・対象・公開日・詳細ページへのリンク** を一覧として抽出する。一覧は公開日の新しい順に並んでいる。直近のもの（過去1〜2か月分）を対象にすれば十分。

- 公開日が「リリース日」列に記載されている。「この文書は、新しい情報が…」のような注記行はリリースではないので除外する

### 2. 重複チェック（新着リリースの特定）

`content/catchup/` ディレクトリ内の既存の `apple-security-releases-*.md` ファイルを確認し、すでに記録済みのリリース（名称＋公開日／詳細ページURL）を把握する。

```bash
ls content/catchup/apple-security-releases-*.md 2>/dev/null
```

一覧のリリースのうち、既存ファイルにまだ含まれていないもの（詳細ページURL、なければ名称＋公開日が未掲載のもの）を「新着」とする。

- 新着が1件もない場合は、ファイルを作成せず「新しいセキュリティリリースはありませんでした」と報告して終了する
- 既存ファイルが1つもない場合（初回）は、直近1〜2か月分を新着として扱う

### 3. 各リリースの要約作成

新着リリースそれぞれについて、可能なら詳細ページを取得し、本文に基づいて日本語で2〜3行の要約を作成する。

- 対象製品・バージョン、対象デバイス（Available for）、修正された脆弱性の概要や件数、注目すべきCVE（ゼロデイ・実環境で悪用されたもの等）があれば簡潔にまとめる
- 詳細ページが取得できなかった場合は、一覧ページの情報をもとに要約する（推測で内容を作らない）

### 4. Markdownダイジェストの作成

新着リリースを以下のフォーマットで **1つのファイル** にまとめる。

**ファイル名**: `apple-security-releases-YYYY-MM-DD.md`（YYYY-MM-DD は実行日）
**保存先**: リポジトリの `content/catchup/` ディレクトリ

**テンプレート**:

```markdown
# Apple セキュリティリリース キャッチアップ: YYYY-MM-DD

> 取得日: YYYY-MM-DD
> ソース: [Apple セキュリティリリース](https://support.apple.com/ja-jp/100100)

## 今回の注目ポイント

[新着リリースのうち特に重要な点（実環境で悪用された脆弱性、広く影響するOS更新など）を3〜5行でまとめる]

---

## リリース一覧

### [名称（製品・バージョン）]
- **公開日**: YYYY-MM-DD
- **対象**: [Available for に記載のデバイス/OS]
- **詳細**: [詳細ページのURL]
- **要約**: [2〜3行で日本語要約。修正された脆弱性の概要・注目CVEなど]

（新着リリースごとに上記ブロックを繰り返す。公開日の新しい順に並べる）
```

新着リリースをすべて漏れなく含める。名称はページのものをそのまま使う。

このスキル単体ではpushを行わない。push まで自動化したい場合は `frontend-catchup-and-push` スキルを使う。

### 5. 定期実行について

このスキルは `frontend-catchup-and-push` スキルのステップに組み込まれており、`schedule` スキルによる定期キャッチアップの一部として自動実行される。Apple セキュリティリリースは不定期（OS更新時に集中）なので、週1回のスケジュールが適切。

## よくある失敗と対処

| 失敗 | 対処 |
|------|------|
| ページにアクセスできない（ブロック） | 環境のネットワークポリシーで `support.apple.com` が許可されているか確認。取得不可ならこのソースだけスキップして報告 |
| RSSフィードを探して時間を浪費する | このソースに公式RSSはない。一覧ページを直接取得する |
| 記憶でリリース内容を補完してしまう | 取得できた内容のみ使う。取得失敗時はスキップ |
| 既存リリースまで重複してダイジェストに入れる | ステップ2の重複チェックを必ず行う |
| 新着ゼロなのに空ファイルを作る | ファイルを作らず「新しいセキュリティリリースはありませんでした」と報告 |
